# AI 医院导入过程管理系统 MVP 开发任务拆分

## 1. 目标

把产品拆成可以直接排期的前后端开发任务，默认目标是做一个单体 Web MVP。

## 2. 里程碑

### M1: 基础底座

- 项目初始化
- 数据库建表
- 登录和基础权限
- 基础字典配置

### M2: 问询闭环

- 聊天入口
- AI 结构化提取
- 纪要入库
- 自动生成任务

### M3: 管理闭环

- 医院台账
- 任务追踪
- 管理看板

### M4: 可用性增强

- 筛选和搜索
- 审核 / 修正抽取结果
- 提醒和逾期标记

## 3. 后端任务

### 3.1 基础工程

- 初始化 Node.js 服务框架
- 接入 PostgreSQL
- 建立迁移机制
- 建立环境配置和日志
- 建立统一错误处理

### 3.2 用户与权限

- 用户表读写
- 登录接口
- 基础角色权限
- 用户与区域绑定

### 3.3 基础配置

- 区域 CRUD
- 医院 CRUD
- 科室 CRUD
- 项目阶段 CRUD
- 问题标签 CRUD

### 3.4 医院项目模块

- 创建医院项目
- 获取项目列表
- 获取项目详情
- 更新项目阶段 / 状态
- 更新项目摘要

### 3.5 聊天与 AI

- 创建聊天会话
- 保存聊天消息
- 调用模型进行结构化提取
- 落库 `project_updates`
- 处理抽取失败重试

### 3.6 任务模块

- 根据抽取结果生成任务
- 任务列表查询
- 更新任务状态
- 逾期任务识别

### 3.7 看板统计

- 阶段分布统计
- 超期未推进医院统计
- 逾期任务统计
- 问题标签分布统计
- 区域维度统计

## 4. 前端任务

### 4.1 基础框架

- 项目初始化
- 路由
- 登录态管理
- API 请求封装
- 全局布局

### 4.2 聊天页

- 聊天消息列表
- 输入框与发送
- AI 提取结果预览
- 提交本次纪要

### 4.3 我的任务页

- 我的任务列表
- 按状态筛选
- 更新任务状态

### 4.4 医院列表页

- 医院项目列表
- 按区域筛选
- 按阶段筛选
- 搜索医院

### 4.5 医院详情页

- 基础信息卡片
- 当前状态卡片
- 任务列表
- 历史时间线
- 问题标签展示

### 4.6 管理看板页

- KPI 卡片
- 阶段分布图
- 卡住医院列表
- 高频问题列表

### 4.7 配置页

- 医院维护
- 区域维护
- 阶段维护
- 标签维护

## 5. AI 任务

### 5.1 提示词设计

- 设计问询提示词
- 设计结构化抽取提示词
- 设计摘要生成提示词

### 5.2 输出约束

- 定义 JSON schema
- 后端做 schema 校验
- 失败时回退到人工确认

### 5.3 数据质量

- 医院名识别和映射
- 联系人识别
- 标签归类
- 时间字段标准化

## 6. 建议分工

### 后端工程师

- 数据建模
- API
- AI 调用与抽取落库
- 统计接口

### 前端工程师

- 聊天页
- 台账页
- 任务页
- 看板页

### 产品 / 业务负责人

- 定义项目阶段
- 定义必填字段
- 定义管理指标
- 给出真实问询样本

## 7. 第一版最关键的验收点

- 一次聊天能生成一条有效推进记录
- 推进记录能挂到正确医院项目下
- 能自动生成任务
- 管理者能看到卡点医院
- 管理者能看到逾期任务

## 8. 最小迭代顺序

### Sprint 1

- 数据库表结构
- 登录
- 医院 / 区域 / 阶段基础数据

### Sprint 2

- 聊天页
- 聊天消息保存
- AI 结构化抽取

### Sprint 3

- 项目更新入库
- 自动任务生成
- 医院详情页

### Sprint 4

- 任务页
- 看板页
- 统计接口

## 9. 最容易延期的点

- 医院项目阶段定义不清
- AI 提取字段频繁变化
- 想同时做多个聊天入口
- 没有先固定数据结构就开始写页面

## 10. 建议开发顺序结论

正确顺序：

1. 先定数据模型
2. 再做 AI 输出结构
3. 再做聊天入口
4. 再做台账和任务
5. 最后做看板

错误顺序：

1. 先做一个很花的聊天 UI
2. 后面再想怎么落库
3. 最后发现医院台账根本串不起来
